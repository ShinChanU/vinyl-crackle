# 00. Kickoff — 아이디어에서 스펙까지

## 아이디어의 시작

LP 턴테이블로 음악을 들으면 특유의 "지지직" 소리가 난다.
이걸 영어로는 **crackle** 또는 **surface noise**라고 부른다.

유튜브 뮤직, 스포티파이 같은 스트리밍 서비스로 음악을 들을 때,
이 크래클이 자동으로 입혀지면 어떨까? 라는 생각에서 출발했다.

## 기술적 접근 방식 선택

### 어디서 오디오를 가로채는가?

스트리밍 서비스에 바이닐 모드를 오버레이하려면, 유저의 환경에 따라 진입점이 다르다:

| 유저 환경 | 진입점 | 난이도 |
|-----------|--------|--------|
| PC 브라우저 | 크롬 확장 프로그램 | 낮음 |
| PC 데스크톱 앱 | 가상 오디오 드라이버 | 높음 |
| Android | AudioPlaybackCapture API | 중간 |
| iOS | 불가능 (자체 플레이어만 가능) | 매우 높음 |

**결정: 크롬 확장 프로그램부터 시작한다.**
이유: 만들기 가장 쉽고, "사람들이 진짜 쓸까?"를 가장 빠르게 검증할 수 있다.

### Content Script vs tabCapture

크롬 확장에서 오디오를 잡는 방법이 두 가지 있다:

**Option A: `chrome.tabCapture`**
- 탭 전체 오디오를 캡처
- Manifest V3에서 offscreen document 필요
- 모든 오디오를 잡을 수 있지만 셋업이 복잡

**Option B: Content Script + `createMediaElementSource`**
- 페이지의 `<audio>`, `<video>` 요소를 직접 Web Audio API에 연결
- 더 직관적이고 퍼미션이 가벼움
- 단, 이미 AudioContext에 연결된 요소에는 중복 연결 불가

**결정: Content Script 방식을 채택한다.**
이유: 퍼미션이 가벼워 유저 신뢰도가 높고, 구현이 단순하다. 실패하면 tabCapture로 전환한다.

### 언어 선택: Vanilla JS vs TypeScript

처음엔 빌드 없이 Vanilla JS로 가려 했지만, TypeScript로 결정했다.

- 타입 안전성: AudioWorklet 파라미터, 메시지 페이로드 등에서 타입 실수 방지
- IDE 지원: 자동완성, 리팩터링이 편함
- esbuild: 빌드가 수 ms로 끝나서 DX 손실이 거의 없음
- 향후 확장: Rust WASM으로 DSP 부분만 교체할 때 인터페이스가 명확함

### 패키지 매니저: pnpm

npm 대비 빠르고 디스크 효율적. 프로젝트 규모와 관계없이 pnpm이 합리적 기본값.

## 크래클 노이즈의 구성

실제 LP 크래클은 3가지 요소의 합이다:

1. **Surface Noise** — 바늘이 홈을 따라가며 내는 연속적 "쉬~" 소리. 필터링된 화이트 노이즈로 시뮬레이션.
2. **Clicks & Pops** — 먼지나 흠집에 의한 단발성 "톡, 탁" 소리. 랜덤 임펄스로 생성.
3. **Dust Particles** — 먼지 입자가 바늘에 닿을 때의 짧은 "치직" 소리. 짧은 노이즈 버스트.

이 3가지를 독립적으로 조절할 수 있게 하고, 프리셋으로 묶어서 제공한다.
